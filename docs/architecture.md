# Architecture

This document describes how Moonraker is laid out internally so contributors can extend it without reverse-engineering the existing code.

## Goals

1. **POSIX-conformant applets.** Each applet implements the common flags, edge cases, and exit-code conventions of its system equivalent.
2. **Single static binary.** Distribute one file per platform via `luastatic`.
3. **Cross-platform.** Designed for Linux, macOS, and Windows; CI workflows target all three.
4. **Easy to extend.** Adding an applet is drop-in: one file, one line in the manifest, one spec.

## Module responsibilities

| Module | Purpose |
|--------|---------|
| [`src/main.lua`](../src/main.lua) | Entry point. Loads applets and hands off to the dispatcher. |
| [`src/cli.lua`](../src/cli.lua) | Multi-call dispatch (argv[0] basename match) and wrapper dispatch (`moonraker <applet>`). Intercepts `--help`, `--version`, `--list`. |
| [`src/registry.lua`](../src/registry.lua) | Holds the name → applet map. Aliases share the same applet entry. |
| [`src/common.lua`](../src/common.lua) | Shared helpers: error formatting, path helpers, platform detection. |
| [`src/version.lua`](../src/version.lua) | Single source of truth for the version string. |
| [`src/usage.lua`](../src/usage.lua) | Long-form help text per applet, keyed by name. |
| [`src/applets/init.lua`](../src/applets/init.lua) | Manifest. Required by `main.lua` to load every applet into the registry. Regenerated by `lua build.lua --regen-only`. |
| [`src/applets/<name>.lua`](../src/applets/) | One applet per file. Returns `{name, aliases, help, main}`. |
| [`src/vendor/`](../src/vendor/) | Pure-Lua dependencies — `re.lua` (LPeg's regex frontend) and `sha2.lua` (pure-Lua hash library). |
| [`src/cdeps/`](../src/cdeps/) | Vendored C source compiled into the binary — `lpeg/`, `zlib/`, `bzip2/`, `luasocket/`. |
| [`src/regex.lua`](../src/regex.lua) | POSIX ERE → LPeg compiler used by `sed` and `awk`. |
| [`src/hashing.lua`](../src/hashing.lua) | Shared logic for the `*sum` applet family. |
| [`src/socket.lua`](../src/socket.lua) | LuaSocket helper bundled at the top level so `require("socket")` resolves cleanly. |

## Applet shape

Every applet is a self-contained Lua module that returns this table:

```lua
return {
  name = "<applet-name>",
  aliases = {},                      -- optional, defaults to empty
  help = "<one-line summary>",
  main = function(argv) return rc end,
}
```

`argv` is a Lua-style table where `argv[0]` is the canonical applet name (after symlink resolution and suffix stripping) and `argv[1..n]` are the user's positional arguments. `main` returns an integer exit code.

## Dispatch flow

1. The OS runs the binary. Control reaches [`src/main.lua`](../src/main.lua).
2. `applets.load_all()` registers every applet in the manifest.
3. `cli.main(arg)` runs:
   - Compute `program_stem` from `arg[0]` (basename, lowercased, `.exe`/`.lua` stripped).
   - If `program_stem` matches a registered applet (and isn't `moonraker`), dispatch directly — **multi-call mode**.
   - Otherwise, treat `arg[1]` as the applet name and `arg[2..n]` as its arguments — **wrapper mode**.
4. The dispatcher intercepts `--help` (long form) before invoking the applet. It deliberately does *not* intercept `-h`, since several POSIX applets (`df -h`, `du -h`, `sort -h`) overload it.

## Adding an applet

1. **Write the module.** Create `src/applets/<name>.lua` exporting `{name, aliases, help, main}`. Keep it self-contained — shared helpers go in [`src/common.lua`](../src/common.lua).
2. **Register it.** Either add the line manually to [`src/applets/init.lua`](../src/applets/init.lua), or run `lua build.lua --regen-only` to regenerate the manifest from the contents of `src/applets/`.
3. **Add long-form help.** Drop a usage block keyed by your applet's name into [`src/usage.lua`](../src/usage.lua). The dispatcher renders it after the one-line summary when the user passes `--help`.
4. **Test it.** Add `spec/applets/<name>_spec.lua` using the [`helpers`](../spec/helpers.lua) module. Cover at least: each documented flag, edge cases, and any error path that produces a specific exit code or stderr message.
5. **Document it.** Add a long-form usage block to [`src/usage.lua`](../src/usage.lua) and add the new applet to [`README.md`](../README.md)'s applet table.

## I/O model

Applets write to `io.stdout` and `io.stderr` directly. This keeps applet code small and avoids passing handles through every call site. The test helper ([`spec/helpers.lua`](../spec/helpers.lua)) replaces `io.stdout` and `io.stderr` with in-memory buffers that share the file-handle interface, so tests need no per-applet hooks.

### Binary-safe I/O

Lua strings are byte sequences, so `io.stdout:write(s)` writes the bytes verbatim — except on Windows, where `stdout` defaults to text mode and translates `\n` to `\r\n`. Applets that must round-trip binary content (`cat`, `gzip`, `dd`, `tee`) need to set stdout/stdin to binary mode. The mechanism for that is a Phase 6 concern; for now, the trivial applets emit text-only.

## Build

[`build.lua`](../build.lua) discovers applets in `src/applets/`, regenerates the manifest, and invokes `luastatic` from inside `src/` so module names resolve correctly. The resulting binary embeds every Lua module by name; `require("applets.echo")` resolves to the embedded copy at runtime.

### C dependencies (`src/cdeps/`)

For each subdirectory of [`src/cdeps/`](../src/cdeps/), the build compiles every `.c` file with `gcc -c -O2 -I<lua-include> -I<lib-dir>` and passes the resulting `.o` files to `luastatic`. luastatic runs `nm` on each `.o`, looks for a `luaopen_<name>` symbol, and registers any matches as embedded Lua C modules. Object files without a `luaopen_*` symbol are linked as supporting code.

A subdirectory may include a `cflags` file (one flag per line) that adds extra compiler flags for that library. zlib uses this to set `-DLZLIB_COMPAT`; bzip2 sets `-D_FILE_OFFSET_BITS=64`.

- [`src/cdeps/lpeg/`](../src/cdeps/lpeg/) — LPeg 1.1.0 (MIT, Roberto Ierusalimschy). `lptree.c` exports `luaopen_lpeg`. After the build, `require("lpeg")` returns the embedded module.
- [`src/cdeps/zlib/`](../src/cdeps/zlib/) — zlib 1.3.1 (zlib license, Jean-loup Gailly + Mark Adler) plus `lua_zlib.c` (MIT, Brian Maher). Backs `gzip`, `gunzip`, `tar -z`, and `zip`/`unzip`.
- [`src/cdeps/bzip2/`](../src/cdeps/bzip2/) — bzip2 1.0.8 (BSD-style, Julian Seward) plus `lua_bzip2.c` (our own ~150-line minimal binding). Backs `tar -j`.
- [`src/cdeps/luasocket/`](../src/cdeps/luasocket/) — LuaSocket 3.1.0 (MIT, Diego Nehab), POSIX TCP/UDP/select sources only. Backs `nc` and `dig`.

### Pure-Lua dependencies (`src/vendor/`)

Each `*.lua` file in [`src/vendor/`](../src/vendor/) is bundled into the binary at the same level as the applets. `require("vendor.<name>")` resolves to the embedded copy.

- [`src/vendor/re.lua`](../src/vendor/re.lua) — LPeg's regex-style frontend, unmodified.
- [`src/vendor/sha2.lua`](../src/vendor/sha2.lua) — Egor Skriptunoff's `pure_lua_SHA` v12 (MIT). Backs the `*sum` applets.

## Testing

[`busted`](https://lunarmodules.github.io/busted/) drives the test suite. Specs live in `spec/` mirroring `src/`:

- `spec/registry_spec.lua` — registry behavior, in isolation.
- `spec/cli_spec.lua` — dispatcher behavior (multi-call vs wrapper, intercepts).
- `spec/applets/<name>_spec.lua` — per-applet behavior.

Tests run the dispatcher in-process with captured stdio (see [`spec/helpers.lua`](../spec/helpers.lua)). This is fast (~70 seconds for the full suite on Linux) and gives each spec deterministic stdout/stderr buffers to assert against.

When a black-box behavioral test would catch what an in-process test misses (CRLF handling, signal propagation, real exit codes), prefer adding a separate integration test that spawns the built binary via `io.popen`.

## CI

[`.github/workflows/ci.yml`](../.github/workflows/ci.yml) runs on every push and PR:

1. **Lint stage** — `luacheck` + `stylua --check`.
2. **Test stage** — `busted` on Linux, macOS, and Windows.
3. **Build stage** — `lua build.lua` on Linux, macOS, and Windows; smoke-test the produced binary; upload it as a build artifact.

[`.github/workflows/release.yml`](../.github/workflows/release.yml) fires on `v*.*.*` tags or manual dispatch, builds per-platform binaries with target-specific filenames, and publishes a GitHub Release.

## Versioning

Moonraker follows [Semantic Versioning](https://semver.org). Breaking changes to applet output, exit codes, or flag semantics are major bumps. New applets, flags, or aliases are minor bumps. Bug fixes are patch bumps. The version string lives in [`src/version.lua`](../src/version.lua) and is also embedded in the binary via the `--version` flag.

## Local-only files

A few paths are excluded from version control via [`.gitignore`](../.gitignore):

- `dist/` — the build output directory.
- `src/cdeps/**/*.o` — intermediate object files from the C-dep build.
- `PLAN.md` — long-form design notes maintained locally.
